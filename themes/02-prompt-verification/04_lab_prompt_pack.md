# 04. Лабораторная работа 1: пакет промптов для типовых задач педагога

## Формат
Индивидуальная работа.

## Цель
Собрать пакет из трех рабочих промптов для типовых образовательных задач, провести минимум 2 итерации улучшения на каждом и получить хотя бы один результат в структурированном формате.

## Разрешенные инструменты
- `GigaChat`
- `YandexGPT`
- локальная модель (`Ollama`, `LM Studio` или аналог)
- другой доступный инструмент по согласованию с преподавателем

Минимальное требование: использовать хотя бы один доступный отечественный или локальный вариант.

## Общий учебный мини-кейс
Во всех заданиях используйте один и тот же кейс.

**Дисциплина:** информатика  
**Аудитория:** 1 курс СПО  
**Тема:** ветвление в алгоритмах  
**Цель:** научить отличать линейный алгоритм от ветвящегося, читать блок-схему с условием и объяснять, почему в простом `if/else` выполняется только одна ветвь.

### Опорный учебный фрагмент
1. Линейный алгоритм выполняет команды последовательно, без выбора ветви.
2. Ветвящийся алгоритм содержит условие и выбирает одну из альтернативных ветвей.
3. Условие обычно записывается как логическое выражение.
4. В блок-схеме проверка условия изображается ромбом.
5. В конструкции `if/else` при истинном условии выполняется одна ветвь, при ложном - другая.

## Три обязательных сценария

| Сценарий | Что нужно получить | Обязательный формат |
|---|---|---|
| 1. Фрагмент занятия | мини-план этапа объяснения новой темы | таблица |
| 2. Оценочные задания | 4 задания с ожидаемыми ответами и критериями | JSON или таблица |
| 3. Персонализированный feedback | короткое письмо студенту по результату мини-проверки | текст по шаблонным блокам |

## Обязательные артефакты
1. Матрица промптов по всем трем сценариям.
2. Не менее 2 итераций улучшения на каждый сценарий.
3. Итоговый approved-пакет из трех промптов.
4. Сравнение результатов `до/после`.
5. Краткая рефлексия: что чаще всего ухудшало ответ модели.

## Порядок выполнения
1. Для каждого сценария написать первый промпт.
2. Получить ответ модели и зафиксировать конкретный дефект.
3. Изменить промпт так, чтобы исправление было связано именно с найденным дефектом.
4. Повторить генерацию.
5. Оценить улучшение по шкале качества.
6. Выбрать итоговый approved-вариант.

## Что считать дефектом
- нет нужного формата;
- модель выходит за рамки темы;
- уровень сложности не соответствует аудитории;
- критерии оценивания неполные;
- feedback слишком общий или не подсказывает следующий шаг;
- в ответе появляются лишние данные или домыслы.

## Шаблон сильного промпта
```text
Ты - [роль].

Контекст:
- дисциплина: информатика;
- аудитория: 1 курс СПО;
- тема: ветвление в алгоритмах;
- цель: отличать линейный алгоритм от ветвящегося.

Задача:
[что именно нужно сделать]

Опорный материал:
1. ...
2. ...
3. ...

Ограничения:
- не придумывай факты за пределами опорного материала;
- не используй реальные персональные данные;
- соблюдай уровень 1 курса СПО;
- объем ответа должен быть компактным.

Формат ответа:
[таблица / JSON / структурированный текст]
```

## Минимальные требования к формату результата
- Для сценария 1 обязательно вернуть таблицу.
- Для сценария 2 обязательно вернуть строгий JSON или таблицу с критериями.
- Для сценария 3 обязательно выделить блоки: `сильные стороны`, `что улучшить`, `следующий шаг`.

## Запрещено
- использовать реальные персональные данные;
- сдавать только финальные ответы без журнала итераций;
- оценивать качество фразой «стало лучше» без указания, что именно улучшилось;
- смешивать все три сценария в один универсальный промпт.

## Самопроверка перед сдачей
1. Есть ли все 3 сценария?
2. Есть ли минимум 2 итерации на каждый сценарий?
3. Есть ли хотя бы один структурированный формат?
4. Понятно ли, какой дефект исправляла каждая итерация?
5. Есть ли итоговый approved-пакет?

## Критерии зачета ЛР1
ЛР1 считается выполненной, если:
1. Заполнена матрица по трем сценариям.
2. Для каждого сценария зафиксировано не менее 2 итераций.
3. Есть хотя бы один результат в формате JSON или таблицы.
4. Есть краткая рефлексия о типовых дефектах и способах их исправления.

Иначе - доработка.

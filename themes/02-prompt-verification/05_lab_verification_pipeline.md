# 05. Лабораторная работа 2: контур верификации AI-артефакта

## Формат
Индивидуальная работа.

## Цель
Проверить один AI-артефакт по полному контуру верификации и принять обоснованное решение `approve/revise/reject` перед использованием результата в учебном процессе.

## Что берем на вход
Используйте один результат из ЛР1:
- либо набор оценочных заданий;
- либо feedback-сообщение;
- либо фрагмент занятия.

Если ЛР1 не завершена, преподаватель выдает готовый демонстрационный AI-ответ для проверки.

## Тот же учебный мини-кейс
Проверка выполняется на основе того же кейса, который использовался в ЛР1.

**Дисциплина:** информатика  
**Аудитория:** 1 курс СПО  
**Тема:** ветвление в алгоритмах  
**Цель:** отличать линейный алгоритм от ветвящегося, читать блок-схему с условием и объяснять логику `if/else`.

### Опорный учебный фрагмент
1. Линейный алгоритм выполняет действия последовательно.
2. Ветвящийся алгоритм содержит условие выбора одной из ветвей.
3. Условие записывается как логическое выражение.
4. В блок-схеме условие показывается ромбом.
5. В простом `if/else` выполняется только одна из двух ветвей.

## Обязательные артефакты
1. Чек-лист проверки.
2. Таблица найденных дефектов и исправлений.
3. Исправленная версия результата.
4. Решение `approve/revise/reject` с кратким обоснованием.
5. Короткая заметка о данных и прозрачности использования ИИ.

## Порядок выполнения
1. Выбрать один AI-артефакт.
2. Сопоставить его с опорным учебным фрагментом.
3. Пройти все поля единого чек-листа.
4. Зафиксировать найденные дефекты.
5. Исправить результат или сделать вывод, что исправление нецелесообразно.
6. Принять финальное решение `approve/revise/reject`.

## Что считается доказательством
- ссылка на конкретный пункт опорного учебного фрагмента;
- указание на несоответствие заявленной цели;
- отсутствие критерия оценивания в самом результате;
- пример лишних данных или небезопасной формулировки;
- признак того, что результат нельзя объяснить студенту или преподавателю.

## Как принимать решение

| Решение | Когда применять |
|---|---|
| `approve` | дефекты отсутствуют или не влияют на учебное применение |
| `revise` | дефекты исправимы без полной переделки результата |
| `reject` | результат не соответствует теме, опасен по данным или непригоден как основа |

## Раздел про данные и прозрачность
Кратко ответьте на два вопроса:
1. Какие данные в этом кейсе действительно нужны модели?
2. Как пользователь узнает, что результат был сгенерирован ИИ и затем проверен человеком?

## Контрольные сценарии, которые обязательно нужно уметь распознать
1. **Корректный учебный текст** - можно принять или незначительно поправить.
2. **Фактическая ошибка** - красивый текст, но с нарушением содержания темы.
3. **Лишние персональные данные** - результат содержит данные, не нужные для решения задачи.

## Запрещено
- проверять результат только по стилю, без сопоставления с опорным материалом;
- ставить `approve`, если обнаружены критические ошибки в фактах или данных;
- ограничиваться общей фразой «модель ошиблась» без конкретного доказательства.

## Самопроверка перед сдачей
1. Заполнены ли все строки чек-листа?
2. Есть ли минимум 3 зафиксированных наблюдения или явное объяснение, почему дефектов меньше?
3. Есть ли исправленная версия результата?
4. Есть ли решение `approve/revise/reject`?
5. Есть ли раздел про данные и прозрачность?

## Критерии зачета ЛР2
ЛР2 считается выполненной, если:
1. Заполнен единый чек-лист по всем слоям проверки.
2. Есть таблица дефектов и исправлений.
3. Есть исправленная версия результата или обоснованный `reject`.
4. Принято и объяснено решение `approve/revise/reject`.
5. Есть краткая фиксация по данным и прозрачности.

Иначе - доработка.

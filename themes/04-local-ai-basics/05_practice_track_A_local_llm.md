# 05. Практика - трек A: реальный запуск локальной LLM и создание учебного артефакта

> Если вы только входите в тему, сначала откройте [01_student_start_here.md](01_student_start_here.md).

## Формат
Индивидуальная работа.

> `A-core` - обязательный контур. Он подтверждает, что студент действительно поднял локальный ИИ-инструмент и получил локальный результат.

## Цель
Реально запустить локальную LLM на своем устройстве, создать один простой учебный артефакт и зафиксировать наблюдаемые ограничения локального маршрута.

## Старт на занятии
После вводной речи преподавателя действуйте по этому короткому маршруту:
1. Выберите локальный маршрут: `Ollama` или `LM Studio`.
2. Для старта выберите **компактную instruct-модель**.
3. Подготовьте рабочую папку для файлов темы.
4. Запустите локальный инструмент и модель.
5. Получите один **короткий** учебный артефакт по одному из шаблонов ниже.
6. Сразу сохраните доказательство запуска.

Если один маршрут не поднимается, переходите на другой и фиксируйте это в отчете.

## Разрешенные маршруты
- `Ollama`
- `LM Studio`

Если у студента не получилось поднять один маршрут, допускается переключение на другой. Сдача без реального запуска не принимается.

## Какую модель брать по умолчанию
- Если не знаете, что выбрать, берите **компактную instruct-модель**.
- Слабая машина: маленькая модель и короткий ответ.
- Средняя машина: компактная или средняя модель.
- Если модель не запускается, переходите на более легкую модель, а не меняйте лабораторную.
- Если выбрали `LM Studio`, по умолчанию работайте через GUI. CLI не обязателен для зачета.

## Общий учебный мини-кейс

**Дисциплина:** информатика  
**Аудитория:** 1 курс ПОО  
**Тема:** двоичная система счисления и перевод чисел  
**Цель:** объяснить, как переводить небольшие десятичные числа в двоичную систему и проверять результат обратным переводом.

## Что нужно сделать
1. Запустить локальную LLM с компактной instruct-моделью.
2. Использовать один из готовых стартовых шаблонов запроса.
3. Получить один короткий учебный артефакт.
4. Сохранить доказательство запуска.
5. Заполнить таблицу наблюдений.

## Разрешенные учебные артефакты
Выберите **один** вариант:
1. Короткое объяснение темы в 5-7 предложениях.
2. Набор из 4 проверочных вопросов.
3. Мини-памятка “Как проверить перевод числа в двоичную систему”.

## Готовые стартовые шаблоны запросов

### Вариант 1: объяснение темы
```text
Ты - преподаватель информатики СПО.

Составь короткое объяснение темы "двоичная система счисления и перевод чисел" для 1 курса ПОО.

Требования:
- 5-7 предложений;
- простой язык;
- один короткий пример перевода;
- без лишней теории и без ссылок на внешние источники.
```

### Вариант 2: проверочные вопросы
```text
Ты - преподаватель информатики СПО.

Составь 4 коротких проверочных вопроса по теме "двоичная система счисления".

Требования:
- 2 вопроса на понимание;
- 2 вопроса на применение;
- после каждого вопроса дай краткий ожидаемый ответ.
```

### Вариант 3: мини-памятка
```text
Ты - помощник преподавателя информатики.

Сделай мини-памятку "Как проверить перевод числа из десятичной системы в двоичную".

Требования:
- не более 6 шагов;
- короткие формулировки;
- в конце один пример проверки.
```

## Обязательные артефакты результата
1. Лог запуска или зафиксированный терминальный/GUI вывод.
2. Скриншот работающего локального инструмента или снимок терминала.
3. Таблица наблюдений.
4. Один учебный артефакт.
5. Краткое сравнение: что локально получилось хорошо и что получилось слабо.

## Таблица наблюдений (заполнить полностью)

| tool | model | device | run_time | observed_limits |
|---|---|---|---|---|
| `Ollama` / `LM Studio` |  |  |  |  |

### Пояснение к полям
- `tool` - выбранный локальный инструмент.
- `model` - название модели.
- `device` - ОС и краткое описание машины.
- `run_time` - время первого запуска или время ожидания результата.
- `observed_limits` - что именно было слабым: скорость, качество, объем, стиль, длина ответа.

## Что считается доказательством реального запуска
- скриншот окна локального приложения;
- фото/скриншот терминала с результатом;
- зафиксированный вывод команды;
- отметка преподавателя, что запуск был показан на защите.

## Запрещено
- использовать только облачный сервис;
- сдавать текст без доказательства локального запуска;
- маскировать отсутствие запуска фразой “модель бы ответила примерно так”;
- использовать реальные персональные данные.

## Самопроверка перед сдачей
1. Есть ли доказательство локального запуска?
2. Понятно ли, какой инструмент и какая модель использовались?
3. Есть ли один завершенный учебный артефакт?
4. Есть ли честно описанные ограничения?
5. Ясно ли, почему этот результат можно или нельзя использовать на занятии без доработки?

## Критерии зачета по треку A
Зачет ставится, если:
1. Локальный запуск действительно выполнен.
2. Есть все обязательные артефакты результата.
3. Таблица наблюдений заполнена полностью.
4. В отчете зафиксированы реальные ограничения локального маршрута.

Иначе - доработка.

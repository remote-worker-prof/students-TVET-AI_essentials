# 06. Практика - трек B (вариативный): прикладной локальный сценарий

> Если вы только входите в тему, сначала откройте [01_student_start_here.md](01_student_start_here.md).

## Суть трека B
После `A-core` студент выполняет **один** прикладной локальный сценарий.

Цель трека B - показать, что локальный контур нужен не только “для запуска модели”, но и для решения конкретной учебной задачи.

## Выберите один из 2 вариантов
1. Локальная транскрибация короткого аудиофрагмента.
2. Локальная работа с документом или раздаткой.

## Ограничение на сложность
Трек B должен оставаться базовым:
- короткий аудиофрагмент;
- короткий документ;
- минимум настроек;
- одна понятная учебная задача.

Трек B не превращается в проект по full RAG, серверной инфраструктуре или продвинутой мультимодальности.

## Вариант 1: локальная транскрибация

### Задача
Взять короткий аудиофрагмент (примерно 30-90 секунд), расшифровать его локально, исправить ошибки и показать, как такой результат можно использовать на занятии.

### Рекомендуемые инструменты
- `faster-whisper` - рекомендуемый путь;
- `whisper.cpp` - допустимый альтернативный путь.

### Обязательные артефакты
1. Исходный короткий аудиофайл.
2. Сырой транскрипт.
3. Исправленный фрагмент.
4. Таблица типовых ошибок распознавания.
5. Короткая заметка о педагогическом применении.

### Таблица ошибок распознавания

| № | raw_fragment | corrected_fragment | error_type | comment |
|---|---|---|---|---|
| 1 |  |  |  |  |
| 2 |  |  |  |  |

### Что написать в заметке о педагогическом применении
- где такой маршрут полезен преподавателю;
- почему автоматический транскрипт нельзя использовать без проверки;
- что в локальном маршруте удобно, а что неудобно.

## Вариант 2: локальная работа с документом

### Задача
Взять **заранее выданный преподавателем** короткий локальный источник (`txt`, небольшая раздатка, короткий `pdf`) и провести мини-сценарий “вопросы к документу”, не выходя за пределы источника.

Для этого варианта в теме 04 используется **один точный маршрут**: работа с документом через `LM Studio` в графическом интерфейсе.

Если у вас нет готового `LM Studio`, не пытайтесь на ходу перестраивать лабораторную в CLI-сценарий. В таком случае выберите вариант 1 с транскрибацией.

### Чем вы будете пользоваться
- `LM Studio` desktop GUI;
- короткий локальный `.pdf` или `.txt`, который можно проверить вручную;
- компактная instruct-модель, уже загруженная в приложение.

### Пошаговый маршрут
1. Возьмите короткий локальный файл, который выдал преподаватель.
2. Откройте `LM Studio` и убедитесь, что компактная instruct-модель уже загружена и готова к работе.
3. Создайте новый чат.
4. Прикрепите к чату ваш `.pdf` или `.txt`.
5. Вставьте в чат стартовый запрос:

```text
Отвечай только по прикрепленному источнику. Если в источнике нет данных, так и напиши: "Недостаточно данных в источнике".
```

6. Задайте **ровно 3 вопроса**, на которые можно ответить по источнику.
7. Для каждого ответа заполните таблицу `question / answer / supported_or_unsupported / evidence_in_source`.
8. Затем задайте **1 вопрос, на который источник не отвечает**.
9. Зафиксируйте, что система не должна уверенно выдумывать ответ, если в источнике нет данных.
10. Сохраните скриншот, где видно чат и прикрепленный документ.

### Что считается правильным выполнением
- вы работаете с коротким локальным источником, а не с длинным набором файлов;
- каждый из 3 основных вопросов можно проверить по тексту вручную;
- в колонке `evidence_in_source` есть понятная опора на источник:
  - короткая цитата;
  - номер страницы;
  - заголовок раздела;
  - фрагмент текста;
- неподдержанный вопрос действительно не имеет ответа в источнике.

### Обязательные артефакты
1. Локальный источник.
2. Скриншот, где видно чат и прикрепленный документ.
3. Три вопроса к документу.
4. Ответы системы.
5. Таблица `supported / unsupported` с evidence.
6. Один вопрос, на который система отвечать не должна.
7. Короткая заметка о педагогическом применении.

### Таблица проверки ответов

| question | answer | supported_or_unsupported | evidence_in_source |
|---|---|---|---|
|  |  | `supported/unsupported` |  |
|  |  | `supported/unsupported` |  |
|  |  | `supported/unsupported` |  |

### Минимальное правило
Хотя бы один вопрос должен быть **неподдержан источником**. Студент обязан показать, что локальный маршрут не должен уверенно отвечать на все подряд.

### Как составить 3 проверяемых вопроса
- вопрос должен относиться к содержанию документа, а не к внешним знаниям модели;
- вопрос должен быть коротким и однозначным;
- ответ на вопрос должен находиться в одном месте текста или на одной странице;
- после ответа вы должны суметь показать, откуда именно он взят.

## Что считается хорошим выбором источника
- короткая учебная раздатка, заранее выданная преподавателем;
- фрагмент конспекта;
- небольшой текстовый файл по теме урока;
- компактный PDF, который можно реально проверить вручную.

### Что не подходит
- очень длинный PDF;
- набор из нескольких файлов сразу;
- документ, который студент не может проверить вручную;
- источник, содержащий реальные чувствительные данные.

## Запрещено
- брать очень длинный документ и пытаться превратить лабораторную в большой RAG-проект;
- использовать облачный сервис вместо локального;
- сдавать только финальный ответ без сырого результата;
- использовать реальные чувствительные данные.

## Самопроверка перед сдачей
1. Выполнен ли реальный локальный прикладной сценарий?
2. Есть ли входной файл и скриншот с прикрепленным документом?
3. Можно ли вручную проверить корректность каждого ответа по `evidence_in_source`?
4. Есть ли один вопрос, на который источник не отвечает?
5. Есть ли заметка о том, как применять это в педагогической практике?
6. Понятны ли ограничения выбранного маршрута?

## Критерии зачета по треку B
Зачет ставится, если:
1. Выполнен один из двух прикладных вариантов.
2. Есть все обязательные артефакты выбранного варианта.
3. Результат проверяем вручную.
4. Есть краткая заметка о педагогическом применении и ограничениях.

Иначе - доработка `B-variant`.

# 07. Дебриф и рубрика оценивания

## Формат итоговой защиты
Студент представляет:
1. Результат `A-core`: локальный запуск LLM, доказательство запуска, учебный артефакт и таблицу наблюдений.
2. Результат `B-variant`: один прикладной локальный сценарий.
3. Краткий вывод: где локальный контур оказался полезен, а где дал заметные ограничения.

## Контур 1: `A-core` (основной зачетный)

| Критерий | Что проверяем | Баллы |
|---|---|---|
| Реальный запуск | Есть ли доказательство, что локальный инструмент действительно работал на устройстве студента | 0-3 |
| Корректный выбор маршрута | Подходит ли выбранный инструмент и модель под возможности устройства | 0-3 |
| Учебный артефакт | Получен ли осмысленный результат для учебной задачи | 0-3 |
| Инженерная аккуратность | Заполнена ли таблица `tool / model / device / run_time / observed_limits` | 0-3 |
| Честная оценка ограничений | Может ли студент объяснить, что локально получилось хорошо, а что слабо | 0-3 |

Максимум по `A-core`: 15 баллов.

## Контур 2: `B-variant` (обязательный прикладной минимум)

| Пункт проверки | Требование | Статус |
|---|---|---|
| Реальный локальный сценарий | Выполнен один из двух вариантов локально | `да/нет` |
| Полный набор артефактов | Есть входной файл, сырой результат и итоговая версия; для документного маршрута есть скриншот с прикрепленным документом | `да/нет` |
| Ручная проверяемость | Можно проверить результат по аудио или источнику; для документного маршрута по каждому ответу есть evidence | `да/нет` |
| Педагогическое применение | Есть краткая заметка о том, где это использовать в учебной практике | `да/нет` |
| Ограничения | Студент явно указывает ограничения выбранного маршрута | `да/нет` |

`B-variant` считается выполненным, если по всем пунктам статус `да`.

## Порог «зачет/незачет» по теме
- **Зачет**: `A-core` >= 10 баллов **и** `B-variant` выполнен.
- **Незачет**: `A-core` < 10 баллов **или** `B-variant` не выполнен.

## Критические ошибки (автоматическая доработка)
- Для `A-core`:
  - нет доказательства реального локального запуска;
  - вместо локального инструмента использован облачный сервис;
  - таблица наблюдений отсутствует или заполнена формально;
  - ограничения локального маршрута не отражены.
- Для `B-variant`:
  - отсутствует входной локальный файл;
  - невозможно проверить результат вручную;
  - прикладной сценарий по сути не был выполнен локально;
  - для документного маршрута нет скриншота с прикрепленным документом или нет evidence по ответам;
  - нет заметки о педагогическом применении.

## Чек-лист преподавателя на защите
- Может ли студент показать, что запуск был реальным, а не описанным “в теории”.
- Понимает ли студент, чем локальный контур отличается от облачного.
- Может ли студент объяснить, почему выбрал именно этот маршрут.
- Видит ли студент ограничения модели и устройства.
- Связывает ли студент технический результат с педагогической задачей.
- Если выбран документный трек, может ли студент показать, откуда в источнике взят каждый поддержанный ответ.

## Рефлексия студента (кратко)
1. Что в локальном маршруте оказалось самым полезным.
2. Что оказалось самым слабым местом локального запуска.
3. В какой реальной педагогической ситуации такой контур стоит использовать, а в какой лучше не стоит.

## Рекомендуемая структура сдачи
- Один документ по `A-core`.
- Один документ по `B-variant`.
- Приложение: скриншоты, лог запуска, входной файл и итоговый результат.
